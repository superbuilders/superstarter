# 18 Seconds — Architecture Plan

This is the single-document overview of how 18 Seconds is structured. It is the document an engineer reads first, before opening the SPEC. Where the SPEC has file paths and column definitions, this document has component boundaries, data flow, and sequencing. The decisions referenced here are recorded in `docs/design_decisions.md`; the file-level realisation is in `docs/SPEC.md`.

## What we're building

A web app that helps adults pass the Criteria Cognitive Aptitude Test (CCAT). The product distinguishes itself from competitor question-banks by being a **mastery engine**: it tracks per-sub-type performance, generates new practice items on demand, and trains the strategic skill of triaging questions at the 18-second mark. **v1 is text-only**: 14 sub-types (5 verbal + 9 numerical). The PRD is the product source of truth; this architecture delivers it within a 2-week build window using the Superbuilder superstarter scaffold.

## System decomposition

The application has six logical components, each with a clear responsibility and a narrow API surface to the others.

**1. Auth & gating.** Auth.js v5 + Google OAuth, wired to Drizzle via a thin shim that converts `Date ↔ epoch-ms` so every Auth.js timestamp lands as a `bigint(_ms)` per the codebase convention. Middleware protects every route except `/api/auth/*`, `/login`, `/api/health`, and the cron endpoints. An `(app)/layout.tsx` gate enforces "user has a completed non-abandoned diagnostic" before any practice route renders; if absent, redirect to `/diagnostic`.

**2. Item bank.** PostgreSQL via Drizzle. `items` carries a discriminated `body` JSON (a single `{ kind: 'text', text }` variant today; the discriminated-union shape exists so additional variants can be added without a schema rewrite), a uniform `options` array of `{ id, text }`, a difficulty tier, and a status (`live`, `candidate`, `retired`). Each option `id` is an opaque 8-character Crockford-base32 string assigned server-side at ingest time; display letters A/B/C/D/E are computed from position-in-array at render time and never stored. The opaque-id design is what unlocks future per-session option shuffling and click-to-highlight in post-session review without breaking explanation cross-references when option order is permuted. v1 has no image storage — items are fully text-based — so there is no S3, no signed-URL minting, and no image route handler. The `pgvector` extension on `items.embedding` powers the validator's uniqueness check.

Explanations are stored in two places that stay in lockstep: `items.explanation` is the rendered prose the user reads, and `items.metadata_json.structuredExplanation` is the canonical structured form (`{parts: [{kind, text, referencedOptions}]}`, length 2 or 3, ordered recognition → elimination → optional tie-breaker). Prose is deterministically rendered from structure; structure is the source of truth. The structure unlocks future click-to-highlight in post-session review (Phase 5/6), where tapping a part highlights the option ids it references via `referencedOptions`. See `docs/plans/opaque-option-ids-and-pipeline-split.md` for the full design.

**OCR import tooling.** Practice-test screenshots in `data/testbank/` are ingested into the item bank via a three-stage offline pipeline: `scripts/import-questions.ts` (extract + solve+verify when no answer is visible) writes one stage-1 JSON per screenshot to `scripts/_stage1/`; `scripts/generate-explanations.ts` runs the explain pass against those JSONs and POSTs each to `/api/admin/ingest-item`; `scripts/regenerate-explanations.ts` rewrites explanations for already-ingested items when the explain prompt changes. Idempotency at every layer keys on the source-image SHA-256, so re-running is safe. The pipeline plus the original four-pass design (extract / solve / verify / explain via Anthropic tool-use) are documented in `docs/plans/ocr-import-screenshots.md` and `docs/plans/opaque-option-ids-and-pipeline-split.md`.

**3. Focus shell.** A single React 19 client component, `<FocusShell>`, with a `useReducer`-driven state. It owns every timer-and-progress primitive. Three bars stack in the chrome row — the question progression bar at top (equal-width segments, leftmost K filled blue when on question K+1; renamed from the pace track in the focus-shell overhaul commit 3), the per-question time bar in the middle, and the session time bar at the bottom. Both time bars **fill from the left edge with red as time elapses** (commits 4–5; the prior "depleting" framing is gone). The chronometer renders the MM:SS countdown at the top-right. The persistent triage prompt and inter-question card live as overlays. Timers run on `requestAnimationFrame` from `performance.now()`, never `setInterval`. The first item is server-rendered into the page response; subsequent items arrive via the `submitAttempt` server action's return value. Latency anchors on a `<ItemSlot>` mount effect that captures `performance.now()` at first paint of every new item, ending at submit. The triage prompt fires once at 18s, stays visible until the user submits or takes it — there is **no auto-submit at the per-question target**, but **when the session timer reaches zero the shell auto-calls `endSession` and navigates to `/post-session/[sessionId]`** (commit 7); the diagnostic case (no session duration) is exempt. Submit is always enabled — a blank submit records `selected_answer: NULL`, which is a first-class signal in the mastery model, not a UI error. The per-question timer is a stacked pair of bars rather than a single bar — a primary bar that fills 0→100% over the per-question target with a discrete blue→red color flip at half-target, and an overflow bar directly below it that stays empty until target then fills 0→100% red over the second target window — together communicating "approaching deadline" and "past deadline" without the auto-submit hard cutoff. The question progression bar's filled segments color-key to session pace: blue when at-or-ahead of pace, red when `elapsedSessionMs / sessionDurationMs > currentQuestionIndex / targetQuestionCount`, with the diagnostic exempt. Audio is hybrid: synthesized 880 Hz ticks fire at integer seconds in the second half of the per-question target, then a randomly-picked MP3 from `data/sounds/` begins looping at target and continues until advance.

**4. Session engine.** Server actions `startSession`, `submitAttempt`, `endSession`, plus `getNextItem`. `getNextItem` dispatches on the session type's `selectionStrategy`: `adaptive` (drills only), `fixed_curve` (diagnostic, full-length, simulation), or `review_queue` (review). Adaptive is computed every call as a pure function over in-session attempts — no in-memory state survives across serverless invocations. Bank-empty fallback chains live here: brutal drills run `brutal → hard → end`; standard drills run the full ladder. Each attempt records `served_at_tier`, `fallback_from_tier` (nullable), and a `fallback_level` metadata field.

**5. Generation pipeline.** A four-stage server module — `generateItem → validateItem → scoreItem → deployItem` — orchestrated as a Vercel Workflow with each stage as a `'use step'` for independent retry. Generator: Claude Sonnet 4 emitting structured JSON matching the per-sub-type Zod template schema (templates emit `body: { kind: 'text', text }`). Validator: GPT-4o, returns 1–5 confidence per check (correctness, ambiguity, difficulty, novelty); pass = all four ≥ 4 AND nearest-neighbor cosine similarity < 0.92. Scorer: weighted sum of validator confidences (no per-option distractor scoring — redundant with the ambiguity check). Deployer: writes the item with `status='candidate'`, embedding included. Bank-target management is visible to the admin via a 14 × 4 grid showing live/candidate/target per cell; top-up is one-click.

**6. Mastery & review.** `computeMastery({ source: 'diagnostic' | 'ongoing' })` is a pure function over the user's last-10 attempts on a sub-type, parameterized so the diagnostic derives a meaningful first-day signal (3-attempt threshold, 1.5× latency relaxation, no `mastered` allowed). After every session, `masteryRecomputeWorkflow` recomputes only sub-types touched in that session. The spaced-repetition queue uses an SM-2 ladder (1/3/7/21 days). The strategy library has 3 strategies per sub-type, differentiated by failure-mode kind (recognition / technique / trap). Full-length post-session reviews include a 30-second strategy gate that picks deterministically (lowest accuracy → highest median latency → lexicographic id; least-recently-viewed strategy via `strategy_views`).

## User journey data flow

1. **Sign in.** Google OAuth via Auth.js. `users` row created on first sign-in.
2. **Diagnostic gate.** `(app)/layout.tsx` checks for a completed non-abandoned diagnostic; if none, redirect to `/diagnostic`. The diagnostic is **untimed at the session level** — it measures capacity, not triage. Items are sampled from the hand-tuned mix in `src/config/diagnostic-mix.ts` (no brutal items; current allocation is 46 entries, **PROVISIONAL** pending the testbank-re-extraction round's empirical re-balance — see PRD §4.1 for the breakdown).
3. **Diagnostic completion.** `endSession` triggers `masteryRecomputeWorkflow` with `source='diagnostic'`. The post-session review captures target percentile and target date inline, then redirects to the Mastery Map.
4. **Mastery Map.** Server-rendered. 14-icon grid (verbal: BookOpen × 5; numerical: Calculator × 9; fill state per mastery level), single-line near-goal text, one primary CTA naming the recommended next session, 30-day rolling triage adherence in low-contrast periphery. When a user lands on `/` immediately after the diagnostic and `mastery_state` hasn't been populated yet (the brief async window between `endSession`'s workflow trigger and the upserts), the page renders a `<ComputingState>` pane that polls via `router.refresh()` and transitions to the populated grid as soon as the workflow lands.
5. **Drill or test launch.** A `/drill/[subTypeId]` configure page captures timer mode and length (defaults to last-time for this sub-type). NarrowingRamp follows (skippable). At session start, the recency-excluded item set (last 7 days) is materialized into `practice_sessions.recency_excluded_item_ids: uuid[]`. Server-rendered first item.
6. **Practice loop.** Each `submitAttempt` writes to `attempts`, recomputes adaptive tier (drills) or reads from the fixed curve, returns the next item or `undefined` on completion. The client `sendBeacon`s a heartbeat every 30 seconds plus a `pagehide` "leaving" signal.
7. **Session end.** `endSession` triggers `masteryRecomputeWorkflow` and `reviewQueueRefreshWorkflow`. User lands on `/post-session/[sessionId]`. Full-length tests gate dismissal behind a 30-second strategy review.
8. **Background work.** `/api/cron/abandon-sweep` runs every minute; `/api/cron/candidate-promotion` runs nightly at 04:00 UTC (shadow mode for 30 days, then enforces).

## Deployment topology

Local development uses a `pgvector/pgvector:pg18` Docker container (Postgres 18 ships `uuidv7()` as a built-in function, which the schema relies on for time-sortable primary keys). Production runs on Vercel with AWS RDS via OIDC federation. LLM access is server-side only — no client SDKs, no streaming. Secrets (`AUTH_GOOGLE_*`, `AUTH_SECRET`, `ANTHROPIC_API_KEY`, `OPENAI_API_KEY`, `CRON_SECRET`) flow through T3 Env. Production lands as a Vercel **preview deployment first**; promotion to production is a one-click move at launch. v1 has no S3 — there are no images to store.

## Major risks and mitigations

**Vercel serverless drops in-memory state.** Every "session-state-like" thing is derived from the database — adaptive tier recomputed per call, recency-excluded set materialized at session start, abandon detection via heartbeats + cron sweep.

**Candidate items with bad validator calls contaminate the bank.** Candidate-promotion shadow mode for 30 days, wide initial bands, soft-archive `retired` status (recoverable via `UPDATE`), and structured 1–5 validator confidences with full reason logs that make rejections debuggable.

**Latency drift undermines the mastery model.** Server-rendered first item so first paint is the question becoming visible, latency anchored on the `<ItemSlot>` mount effect, `requestAnimationFrame` for timer ticks.

**LLM spend runs away.** Pino-logged `tokens_in/tokens_out/cost_estimate_usd` per call; admin dashboard compares today's rate against a 7-day same-hour baseline and surfaces a soft warning at 2× baseline. No auto-pause.

## Build sequencing

**Phase 1 — Foundations (week 1, days 1–3).** Auth + Drizzle adapter shim + complete schema (auth, catalog, practice, review, plus `candidate_promotion_log`, `strategy_views`, `mastery_state.was_mastered`, all `bigint(_ms)`-shaped). pgvector extension. Configuration files (sub-types, strategies, item templates, diagnostic mix, difficulty curves).

**Phase 2 — Real-item path (week 1, days 3–5).** Admin ingest form + tagger LLM call. Hand-seed 70 real items (5 per sub-type × 14 v1 sub-types) via the form. Then OCR-import additional items from CCAT practice-test screenshots via the four-pass LLM pipeline (extract / solve / verify / explain) at `scripts/import-questions.ts` + `scripts/generate-explanations.ts`. The opaque-option-id schema and the structured-explanation contract both shipped within Phase 2, in lockstep with the OCR pipeline. Embedding-backfill workflow runs separately after seeding (the seed loader skips the workflow trigger; `scripts/backfill-missing-embeddings.ts` populates embeddings out of band). Validates the (single-variant) body discriminator end-to-end. See `docs/plans/ocr-import-screenshots.md` and `docs/plans/opaque-option-ids-and-pipeline-split.md`.

**Phase 3 — Practice surface (week 1, days 5–7) — COMPLETE 2026-05-04.** Focus shell + diagnostic flow + Mastery Map + standard drill mode + heartbeats + abandon-sweep cron. Onboarding capture at end of post-diagnostic. The whole user-facing happy path runs end-to-end against real items. Production-deploy coupling is unblocked; deploy-and-dogfood is the next move before Phase 5.

**Phase 4 — Generation pipeline (week 2, days 1–3).** Generator + validator + scorer + deployer + workflow + admin generation page. Vercel + RDS wired. First end-to-end candidate items land. Cost telemetry surfaces.

**Phase 5 — Engine completeness (week 2, days 3–5; v1 scope tightened 2026-05-04).** Adaptive difficulty (drills) + post-session review surface + click-to-highlight + dojo UI rename + belt indicator + full-length test (no strategy gate). Five surfaces from the original Phase-5 list cut from v1 — spaced-repetition queue + review session, the 30-second strategy-review gate, speed-ramp + brutal drill modes, the question-timer toggle (and the session-timer toggle alongside it), and the NarrowingRamp pre-session protocol. See `docs/plans/feature-roadmap.md` "Cut from v1 2026-05-04" for the cut rationale; `docs/plans/phase5-master-plan.md` for the five-sub-phase v1 carve.

**Phase 6 — Polish & cuts (week 2, days 5–7).** Test-day simulation + history tab + candidate-promotion cron in shadow mode + account deletion. PRD §9 cuts apply if behind: simulation, history detail views, NarrowingRamp's visual-narrowing step. The mastery model, generation pipeline, focus shell, and Mastery Map are non-negotiable.
